为什么要使用 Options 模式？#

开发中会面临这样的场景：一个服务或组件（如 LLM 客户端、数据库连接池）需要支持大量配置参数，但大多数场景下用户只需要默认配置。

以 LLM API 调用为例，我们可能需要配置：

  • Temperature（随机性）
  • MaxTokens（最大长度）
  • Timeout（超时时间）
  • …等等

传统方案#

方案代码示例痛点分析
参数爆炸New(0.9, 1000, 30, true, ...)可读性差：必须记住参数顺序；
兼容性差：新增参数需修改所有调用处。
配置结构体New(Config{Temp: 0.9})零值歧义：传 0 是代表“关闭”还是“使用默认值”？
无法强制验证：需要在文档中说明哪些是必填项。

Options 模式： 它利用 Go 的特性，实现了**“默认值优先，按需定制”**。

实现机制#

Options 模式建立在 Go 语言的三个特性之上：函数类型、闭包和变参。

Step 1: 定义配置结构体#

最佳实践：将配置字段私有化，强制用户通过 Options 修改，防止外部直接篡改。

package llm

type options struct {
    model       string
    temperature float64
    maxTokens   int
    apiKey      string
}

// 默认配置
func defaultOptions() options {
    return options{
        model:       "gpt-4",
        temperature: 0.7,
        maxTokens:   1000,
    }
}

Step 2: 定义 Option 函数类型#

定义一个函数类型，它接受配置对象的指针。

type Option func(*options)

Step 3: 利用闭包（Closure）捕获参数#

WithTemperature 并不直接修改配置，而是返回一个 “修改行为”。

func WithTemperature(t float64) Option {
    // 返回一个闭包，它"捕获"了外部变量 t
    return func(o *options) {
        o.temperature = t
    }
}

func WithModel(model string) Option {
    return func(o *options) {
        o.model = model
    }
}

Step 4: 构造函数与变参应用#

构造函数接收可变数量的 Option，依次执行。

type Client struct {
    opts options
}

func NewClient(apiKey string, opts ...Option) *Client {
    // 1. 初始化默认值
    o := defaultOptions()
    o.apiKey = apiKey // 必填参数通常作为普通参数传递

    // 2. 依次应用 Option
    for _, fn := range opts {
        fn(&o)
    }

    // 3. (可选) 最终一致性校验
    // if o.temperature > 2.0 { ... }

    return &Client{opts: o}
}

实践#

在生产环境中，基础的 Options 模式可能还不够。以下是两个变体。

变体 A：带错误处理的 Options (Robust Options)#

如果用户传入了非法的配置值（例如 Temperature = -1），我们不应该默默接受，也不应该直接 Panic。

// 修改 Option 定义，允许返回 error
type Option func(*options) error

func WithTemperature(t float64) Option {
    return func(o *options) error {
        if t < 0 || t > 2.0 {
            return fmt.Errorf("temperature must be between 0 and 2, got %f", t)
        }
        o.temperature = t
        return nil
    }
}

// 构造函数需要处理错误
func NewClient(opts ...Option) (*Client, error) {
    o := defaultOptions()
    for _, fn := range opts {
        if err := fn(&o); err != nil {
            return nil, err // 快速失败
        }
    }
    return &Client{opts: o}, nil
}

变体 B：命名空间组织 (Namespace Options)#

当配置选项非常多时，大量的 WithXXX 函数会污染包的命名空间。我们可以利用结构体方法来进行组织，还能带来极佳的 IDE 补全体验。

// 定义一个空结构体作为命名空间
type optionBuilder struct{}

// 暴露全局变量
var Options = optionBuilder{}

// 将 With 方法挂载到 builder 上
func (optionBuilder) WithModel(m string) Option {
    return func(o *options) {
        o.model = m
    }
}

// 调用示例：代码补全非常友好
// NewClient(
//     Options.WithModel("gpt-4"),
//     Options.WithTemperature(0.9),
// )

总结#

Options 模式是是处理复杂配置场景的理想选择，平衡了简单性与灵活性：

  1. 声明式配置：代码读起来像自然语言
  2. 向后兼容：添加新配置项不会破坏现有的 NewClient 调用
  3. 渐进式增强：先用默认值快速启动，再按需定制
  4. 零依赖：纯语言特性，无需第三方库
